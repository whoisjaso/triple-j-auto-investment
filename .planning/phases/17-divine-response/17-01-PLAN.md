**chatProfiles.ts** -- Create profile identification utility:

1. Export `ProfileType` = `'provider' | 'skeptic' | 'first_timer' | 'struggler' | 'unidentified'`
2. Export `PROFILE_LABELS` -- bilingual display names for each profile (for optional UI indicator):
   - provider: { en: 'Family-Focused', es: 'Enfocado en Familia' }
   - skeptic: { en: 'Detail-Oriented', es: 'Orientado al Detalle' }
   - first_timer: { en: 'First-Time Buyer', es: 'Comprador Primerizo' }
   - struggler: { en: 'Solution Seeker', es: 'Buscador de Soluciones' }
   - unidentified: { en: 'Getting to Know You', es: 'Conociendote' }
   Note: Labels are customer-friendly, NOT the internal profile names. Never show "Skeptic" or "Struggler" to the user.
3. Define bilingual keyword arrays (see RESEARCH.md Pattern 2 for full lists):
   - FAMILY_SIGNALS: family, familia, kids, hijos, wife, esposa, husband, esposo, children, ninos
   - BUDGET_SIGNALS: afford, budget, tight, money, dinero, pagar, caro, expensive, cheap, barato
   - SKEPTIC_SIGNALS: catch, hidden, scam, too good, trust, honest, real, trampa, estafa, problema
   - FIRST_TIMER_SIGNALS: first car, primer carro, first time, primera vez, teen, learning, new driver, nuevo conductor
   - URGENCY_SIGNALS: need, asap, broke down, urgente, necesito, descompuso, stranded, sin carro
   - RELIABILITY_SIGNALS: reliable, break down, confiable, dependable, last long, duradero
4. Export `identifyProfile(messages: string[]): ProfileType` function:
   - Join all messages, lowercase
   - Check each signal category for keyword matches
   - Require messageCount >= 2 before identifying (return 'unidentified' for < 2 messages)
   - Scoring priority: (1) family + budget/reliability = provider, (2) skepticism signals = skeptic, (3) first-timer signals = first_timer, (4) urgency + budget = struggler, (5) budget + reliability = provider (secondary), (6) unidentified

**divine-chat/index.ts** -- Create Supabase Edge Function:

1. Follow the pattern from existing Edge Functions (unsubscribe/index.ts) for structure and CORS headers.
2. Import `GoogleGenAI` from `'npm:@google/genai@^1.30.0'` (Deno npm specifier).
3. Set CORS_HEADERS allowing origin `*`, headers `authorization, x-client-info, apikey, content-type`.
4. Handle OPTIONS preflight returning 'ok' with CORS headers.
5. Check `GEMINI_API_KEY` from `Deno.env.get()`. If missing, return 503 JSON `{ error: 'AI service not configured' }`.
6. Parse request JSON: `{ message, history, vehicleContext, language, identifiedProfile }`.
7. Validate: message must be string, length 1-500. History capped at 20 messages (last 20).
8. Build system prompt via `buildSystemPrompt(vehicleContext, language, identifiedProfile)`:
   - Dealership info: Triple J Auto Investment, 8774 Almeda Genoa Road, Houston TX 77075, (832) 400-9760, Mon-Sat 9AM-6PM, P171632
   - Vehicle context: year, make, model, price, mileage, status, diagnostics, description, listingType, dailyRate, weeklyRate
   - Language instruction: "RESPOND ENTIRELY IN SPANISH" for es, "Respond in English" for en
   - 9 rules from RESEARCH.md (only discuss given info, concise 2-4 sentences, warm/honest, guide toward action, never invent, never discuss competitors, never make up financing terms, all AS-IS)
   - PCP closing framework via `buildPCPGuidance(profile)` with profile-specific adaptation (see RESEARCH.md Pattern 3 for full prompt text)
9. Build contents array from capped history + new user message. Sanitize each history text to max 500 chars.
10. Call `ai.models.generateContentStream()` with model `gemini-2.5-flash`, systemInstruction, temperature 0.7, thinkingConfig `{ thinkingBudget: 1024 }`.
11. Stream response back as `text/plain; charset=utf-8` using ReadableStream + TextEncoder.
12. On stream error, enqueue fallback text: `\n\n[Chat temporarily unavailable. Call (832) 400-9760]`.
13. Wrap everything in try/catch. On error, return 500 JSON `{ error: 'Service temporarily unavailable' }`.
14. Add rate limiting: track requests per session_id (from request body or header). Max 3 messages per 10 seconds per session. Return 429 if exceeded. Use a simple in-memory Map with TTL cleanup (Edge Functions are short-lived, so this is per-instance rate limiting, not global -- sufficient for abuse prevention).

Create `divineChatService.ts` following the pattern of existing services (retellService.ts, trackingService.ts):

1. Export `VehicleContext` interface:
   ```
   year: number, make: string, model: string, price: number, mileage: number,
   status: string, diagnostics: string[], description: string,
   listingType?: string, dailyRate?: number, weeklyRate?: number
   ```
2. Export `ChatMessage` interface: `{ id: string, role: 'user' | 'model', text: string, timestamp: number }`
3. Export `ChatRequest` interface: `{ message: string, history: { role: string, text: string }[], vehicleContext: VehicleContext, sessionId: string, language: 'en' | 'es', identifiedProfile: string }`
4. Export `sendChatMessage(req: ChatRequest): Promise<Response>` function:
   - Construct URL: `${import.meta.env.VITE_SUPABASE_URL}/functions/v1/divine-chat`
   - POST with headers: Content-Type application/json, Authorization Bearer VITE_SUPABASE_ANON_KEY
   - Body: JSON.stringify the request
   - Return the raw Response object (caller handles streaming via response.body reader)
   - If VITE_SUPABASE_URL is not set, throw Error('Chat service not configured')
5. Export `CHAT_STORAGE_KEY = 'tj_divine_chat'` for localStorage persistence
6. Export `loadChatHistory(vehicleId: string): ChatMessage[]` -- reads from localStorage, returns array for specific vehicle
7. Export `saveChatHistory(vehicleId: string, messages: ChatMessage[]): void` -- saves to localStorage, cap at 50 messages per vehicle, max 5 vehicles stored (LRU eviction)
8. Export `clearChatHistory(vehicleId: string): void` -- removes chat for specific vehicle

The service does NOT handle streaming parsing -- that's the hook's job (Plan 02). This service is a thin fetch wrapper + localStorage persistence.
