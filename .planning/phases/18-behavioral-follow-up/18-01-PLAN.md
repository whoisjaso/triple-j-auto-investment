Build the behavioral follow-up backend: a database queue table for pending re-engagement messages, a Postgres function that detects behavioral triggers from session_events data and enqueues follow-ups with appropriate delays, and a Supabase Edge Function that processes the queue by dispatching SMS (Twilio), email (Resend), and AI voice calls (Retell).

Create `phase-18-migration.sql` with the following sections:

**1. follow_up_queue table:**
```sql
CREATE TABLE public.follow_up_queue (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  lead_id UUID NOT NULL REFERENCES public.leads(id) ON DELETE CASCADE,
  trigger_type TEXT NOT NULL CHECK (trigger_type IN ('browse', 'save', 'abandon', 'voice')),
  channel TEXT NOT NULL CHECK (channel IN ('sms', 'email', 'voice')),
  vehicle_id UUID REFERENCES public.vehicles(id) ON DELETE SET NULL,
  vehicle_year INTEGER,
  vehicle_make TEXT,
  vehicle_model TEXT,
  vehicle_slug TEXT,
  views_7d INTEGER DEFAULT 0,
  message_language TEXT DEFAULT 'en' CHECK (message_language IN ('en', 'es')),
  send_after TIMESTAMPTZ NOT NULL,
  sent BOOLEAN DEFAULT FALSE,
  sent_at TIMESTAMPTZ,
  cancelled BOOLEAN DEFAULT FALSE,
  cancelled_reason TEXT,
  error TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```

**2. Deduplication index:** Partial unique index on `(lead_id, trigger_type) WHERE sent = false AND cancelled = false` to prevent duplicate pending messages.

**3. Processing index:** Index on `send_after WHERE sent = false AND cancelled = false` for efficient queue polling.

**4. RLS policies:** Enable RLS. Admin-only access for all operations (`auth.uid() IN (SELECT id FROM public.profiles WHERE is_admin = true)`). Also add an INSERT policy for service_role (Edge Function uses service_role_key so this is handled implicitly, but document it).

**5. preferred_language column on leads:** `ALTER TABLE public.leads ADD COLUMN IF NOT EXISTS preferred_language TEXT DEFAULT 'en';` — this stores the visitor's browsing language at form submission time. The Phase 18-02 frontend plan wires this.

**6. enqueue_behavioral_follow_ups() function (SECURITY DEFINER):**

This function is called by pg_cron hourly. It scans session_events and leads to detect 4 trigger types and inserts into follow_up_queue. Key rules:
- All queries JOIN `leads` WHERE `phone IS NOT NULL AND phone != ''` AND `status = 'New'`
- Only consider leads created within last 48 hours (avoid re-triggering old leads)
- Check `NOT EXISTS` against follow_up_queue to prevent re-enqueuing

**Tier 1 (Browse):** Find leads whose session has `dwell` events with `vehicle_id` but NO `save_toggle` or `form_open` events in the same session. Pick the vehicle with the highest total `dwell_seconds`. Enqueue as `trigger_type='browse'`, `channel='sms'`, `send_after = NOW() + INTERVAL '24 hours'`.

**Tier 2 (Save):** Find leads whose session has `save_toggle` events. Pick the most recently saved vehicle. Enqueue as `trigger_type='save'`, `channel='sms'`, `send_after = NOW() + INTERVAL '4 hours'`. Also populate `views_7d` from `vehicle_view_counts` for scarcity signal.

**Tier 3 (Abandon):** Find sessions with `form_open` events created between 1-3 hours ago that have a matching lead (phone on file from a prior form submission on the same session_id) BUT no lead created AFTER the form_open event in the same session (indicating abandon, not completion). Enqueue as `trigger_type='abandon'`, `channel='sms'`, `send_after = NOW() + INTERVAL '1 hour'`. Also enqueue a second row with `channel='email'` for dual-channel.

**Tier 4 (Voice):** Find leads with `action_type = 'ask_question'` created in the last 4 hours that have NO follow-up lead with `action_type = 'schedule_visit'` from the same session. Enqueue as `trigger_type='voice'`, `channel='voice'`, `send_after = NOW() + INTERVAL '2 hours'`.

For ALL tiers: populate `message_language` from `leads.preferred_language` (defaults to 'en' if NULL).

**Priority logic:** If a lead qualifies for multiple tiers, highest commitment wins. The partial unique index on `(lead_id, trigger_type)` prevents duplicate entries per tier, and the detection order (voice > abandon > save > browse) with `ON CONFLICT DO NOTHING` means only the first qualifying entry per tier is inserted.

**7. cancel_follow_ups_on_conversion() trigger function:**
When leads.status changes from 'New' to 'Contacted' or 'Closed', UPDATE follow_up_queue SET cancelled=true, cancelled_reason='lead_converted' WHERE lead_id=NEW.id AND sent=false AND cancelled=false. Create AFTER UPDATE trigger on leads table.

**8. pg_cron schedules (wrapped in DO block with pg_cron existence check):**
- `detect_follow_ups`: Run `SELECT enqueue_behavioral_follow_ups()` every hour (`0 * * * *`)
- `process_follow_up_queue`: Call Edge Function via pg_net every 5 minutes (`*/5 * * * *`). Use the same pg_net pattern as Phase 9 notification_queue: `SELECT net.http_post(url, headers, body)` with service_role_key in Authorization header.

Include comments that pg_cron requires Pro plan or manual enable (consistent with Phase 9/16 pattern). Wrap cron creation in `DO $$ BEGIN ... EXCEPTION WHEN OTHERS THEN RAISE NOTICE 'pg_cron not available'; END $$;`.

**2. Main handler (Deno.serve):**
- Create Supabase client with `Deno.env.get('SUPABASE_URL')` and `Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')`.
- Query follow_up_queue: `SELECT *, leads(phone, name, email, preferred_language)` WHERE `sent=false AND cancelled=false AND send_after <= NOW()`, limit 50, order by send_after ASC.
- For each queue item, dispatch based on `channel`:
  - **sms**: call `sendSms(lead.phone, buildSmsBody(item, lang))`, then update row.
  - **email**: call `sendEmail({ to: lead.email, subject, html: buildEmailHtml(item, lang) })`, then update row.
  - **voice**: call `triggerRetellCall(item, lead)`, then update row.
- Mark each item as `sent=true, sent_at=NOW()`. On error, set `error` column and still mark sent (prevent infinite retry, matching notification_queue pattern).
- Return JSON: `{ processed, errors }`.

- **abandon**: "Hola! Guardamos tu informacion sobre el {vehicle}. Continua aqui: {url} o llamanos al (832) 400-9760. Responde STOP para cancelar."

Vehicle URL format: `https://triplejautoinvestment.com/vehicles/{vehicle_slug}`. If no slug, use `https://triplejautoinvestment.com/inventory`.

Call Retell API to initiate an outbound voice call. Use server-side credentials:

POST to `https://api.retellai.com/v2/create-phone-call` with:

{

  "from_number": fromNumber,
  "retell_llm_dynamic_variables": {
    "customer_name": lead.name || "there",
